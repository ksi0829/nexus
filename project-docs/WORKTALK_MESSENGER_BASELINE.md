# ZETA WorkTalk 메신저 기반 분석

작성일: 2026-06-12

## 작업 안전 기준

- Desktop의 기존 `worklog` 원본은 수정하지 않는다.
- WorkTalk 개발은 별도 Git 복사본과 전용 브랜치에서만 진행한다.
- 기존 로그인, 업무일지, 일정, 결재, 고객, A/S 기능은 수정하지 않는다.
- 신규 화면은 `/worktalk` 경로와 WorkTalk 전용 모듈로 구성한다.
- 기존 공통 파일 변경이 필요한 경우 변경 이유와 영향 범위를 먼저 검토한다.
- Supabase SQL은 파일로 먼저 작성하고 검토 후 별도로 적용한다.

## 현재 참고 가능한 기능

- 기존 Supabase Auth 로그인과 `profiles` 사용자 정보
- 조직도와 팀 매핑
- 기존 채팅의 1:1·단체방 UX와 데이터 처리 방식
- Supabase Realtime 메시지 수신 구현 방식
- 사용자별 읽음, 파일 첨부, 메시지 고정, 접속 상태 구현 방식
- 모바일 대화 목록/상세 전환 방식
- 브라우저 Notification API 처리 방식

기존 채팅 데이터와 테이블은 현재 그룹웨어 기능이므로 WorkTalk에서 직접 사용하거나
수정하지 않는다. 코드와 UX를 참고하되 신규 모듈로 다시 구성한다.

## WorkTalk 1차에서 보강할 기능

1. 카카오톡과 유사한 독립형 `/worktalk` 전체 화면
2. 팀 구성원 자동 가입 고정 팀방
3. 전 직원 아이디어 소통방
4. 협업방과 개인방의 최근 대화 중심 목록
5. 메시지 검색과 파일 검색 분리
6. 방별 알림 켜기/끄기
7. 관리자·대표·회장 전체 조회 권한
8. 비활성 계정 접근 차단
9. 서비스 워커와 푸시 구독 기반 PWA 알림
10. 향후 결재방을 위한 `approval` 방 유형

## 현재 구조에서 바로 수정하지 않을 부분

- NEXUS는 그룹웨어 채팅과 분리된 독립 메신저로 운영한다.
- 기존 그룹웨어 채팅 팝업과 `AppFrame` 채팅 진입점은 혼선을 막기 위해 제거한다.
- `/worktalk`은 NEXUS 전용 모듈과 `worktalk_*` 테이블만 사용한다.
- 문서 양식과 일부 사용자/조직 정보는 참고하되, 채팅/결재 흐름은 NEXUS 독립 구조로 유지한다.

## DB 보강 원칙

- 기존 Supabase Auth와 `profiles`만 공유한다.
- 기존 `chat_threads`, `chat_participants`, `chat_messages`, `chat_attachments`
  및 관련 RLS 정책은 변경하지 않는다.
- WorkTalk 데이터는 `worktalk_rooms`, `worktalk_room_members`,
  `worktalk_messages`, `worktalk_message_reads`, `worktalk_files` 등 신규
  테이블로 분리한다.
- 방 유형은 최소 `direct`, `group`, `team`, `idea`, `approval`을 지원한다.
- 팀방과 결재방은 일반 사용자가 참여자를 추가하거나 나갈 수 없도록 한다.
- 대표·관리자의 전체 조회는 화면 우회가 아니라 RLS 또는 보안 함수로 구현한다.
- 메시지와 파일 검색을 위한 인덱스와 검색 RPC를 별도로 설계한다.
- 푸시 구독, 방별 알림 설정, 사용자 비활성 상태를 별도 테이블 또는 기존 프로필과 연결한다.

## 기존 채팅에서 확인한 보안 참고사항

- 현재 `chat_participants` 조회 정책의 `using (true)`는 모든 인증 사용자가 전체
  참여자 행을 읽을 수 있다. 기존 정책은 이번 개발에서 수정하지 않고,
  WorkTalk 신규 테이블에서는 같은 정책을 사용하지 않는다.
- 방 생성과 참여자 추가 권한은 방 유형별로 분리해야 한다.
- 관리자 전체 조회 권한과 일반 참여자 권한을 분리해야 한다.
- 첨부파일은 비공개 버킷과 만료형 signed URL을 계속 사용한다.
- 메시지 수정·삭제 정책은 1차에서 제공하지 않는다.

## 첫 구현 단위

첫 개발 단위의 완료 기준은 다음과 같다.

1. 기존 계정으로 `/worktalk`에 진입한다.
2. 팀방, 개인방, 그룹방 목록을 확인한다.
3. WorkTalk 전용 신규 테이블에서 메시지를 실시간 송수신한다.
4. 읽지 않은 메시지 수가 사용자별로 정확하게 표시된다.
5. PC는 방 목록과 대화가 함께 보이고 모바일은 목록과 상세가 전환된다.
6. 기존 그룹웨어 페이지와 기존 채팅 팝업의 동작에는 변화가 없다.

## 개발 순서

1. WorkTalk 전용 타입, 데이터 계층, 훅 구조 작성
2. 신규 `/worktalk` 읽기 전용 화면과 기존 데이터 연결
3. 메시지 전송과 읽음 처리 연결
4. 팀방 생성 SQL과 RLS 강화안 작성
5. 파일 첨부와 메시지·파일 검색
6. 방별 알림 설정과 PWA 푸시
7. 3~5명 실제 사용자 검증
8. 결재방 자동 생성 기능 연결
